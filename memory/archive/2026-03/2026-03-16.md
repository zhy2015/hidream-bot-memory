## 💡 关键经验总结

### API 响应结构标准
Vivago API 统一返回格式：
```json
{
  "code": 0,
  "message": "Success",
  "result": {
    "sub_task_results": [{"image": "..."}],
    "task_id": "..."
  }
}
```
**NOT** `data.works_result` — 这是常见错误来源

### 字段命名规范
- Wrapper → Agent 通信使用 `image_url` / `video_url`
- Agent 内部使用 `image_path` / `video_path`
- 需要双向兼容处理

### 编码器适配策略
系统 FFmpeg 可能缺少 libx264，优先使用：
1. `-c:v copy` (不重新编码)
2. `libopenh264` (OpenH264)
3. `libsvtav1` (AV1)

### 子代理模式价值
复杂修复任务使用 `sessions_spawn` 隔离执行：
- 避免污染主会话上下文
- 并行处理下载/合成/验证
- 自动推送结果

## 🕳️ 本次迭代坑点总结 (Pitfalls)

### 1. 邮件附件中文名问题
**现象**: QQ邮箱发送中文文件名(`废土微光_最终成片.mp4`)被识别为 `.bin` 文件
**解决**: 改用英文文件名(`wasteland_glimmer_final.mp4`)
**经验**: 所有邮件附件统一使用英文/拼音命名

### 2. 视频 CDN 下载超时
**现象**: Vivago 视频 CDN 无代理下载 20MB+ 视频需 2-3 分钟，导致 120s 超时
**解决**: 
- Composer 增加 VPN 代理配置 (`http_proxy=http://127.0.0.1:7890`)
- 超时时间增加到 300s
**数据对比**:
| 方式 | 时间 | 结果 |
|------|------|------|
| 无代理 | 30s+ | ❌ 超时 |
| VPN 代理 | 3.8s | ✅ 21MB 完成 |

### 3. 架构职责分离
**初始设计**: FFmpeg Wrapper 处理远程 URL 下载 + 合成
**问题**: 下载逻辑耦合在 Wrapper 中，超时无法优雅处理
**优化**: 
- Composer 负责下载（带 VPN 代理）
- FFmpeg Wrapper 只处理本地文件合成
- 职责分离，各自专注

### 4. 视频文件交付渠道
**教训**: QQ 直接发送视频文件限制多（大小、格式识别）
**标准流程**: 视频/大文件统一走 `qqmail` 技能发送到用户邮箱

### 5. 运行时环境依赖
**发现**: 
- Seedream/Kling API 调用不需要 VPN
- 视频 CDN 下载必须 VPN
- Clash 需保持运行状态 (PID 910)
